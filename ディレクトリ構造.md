# ディレクトリ構造

## この文書の目的

このリポジトリは学習用リポジトリなので、コードをどこに置くかも学習対象にします。

Rust では、最初は `src/main.rs` にすべて書いても動きます。しかし、型や関数が増えてくると、1ファイルの中で「どこに何があるか」が分かりにくくなります。そこで、早い段階で役割ごとにファイルを分けます。

## 現在の構造

```text
.
├── Cargo.toml
├── Cargo.lock
├── src
│   ├── main.rs
│   ├── lib.rs
│   └── game
│       ├── mod.rs
│       ├── constants.rs
│       ├── cell.rs
│       ├── player.rs
│       ├── coordinate.rs
│       ├── line.rs
│       └── state.rs
├── やりたいこと.md
└── ディレクトリ構造.md
```

## 各ファイルの役割

### `src/main.rs`

実行プログラムの入口です。

今は初期状態を表示するだけにしています。ゲームのルールや状態表現はここに置かず、`src/game/` 以下に置きます。

理由:

- `main.rs` が大きくなると、実験コードとゲーム本体が混ざる
- 将来、コマンドライン引数や表示処理を足しても、ゲームロジックに影響させにくい
- 「実行する場所」と「ルールを定義する場所」を分けて読める

### `src/lib.rs`

ライブラリとして公開する入口です。

`pub mod game;` と書くことで、`src/game/` 以下のゲームロジックを `main.rs` から使えるようにしています。

理由:

- Rust では、`main.rs` だけだと実行用のバイナリ中心になる
- `lib.rs` を作ると、ゲームロジックをライブラリとして整理しやすい
- 将来、テスト・ベンチマーク・別の実験プログラムから同じロジックを使いやすい

### `src/game/mod.rs`

`game` モジュールの入口です。

ここでは、`constants.rs` や `state.rs` などの子モジュールを読み込み、外から使いやすい型を `pub use` で再公開しています。

理由:

- 外側のコードが `game::state::GameState` のような細かい場所を毎回知らなくて済む
- 内部のファイル配置を変えても、外側の使い方を保ちやすい
- `game` モジュール全体として何を公開しているかが分かりやすい

### `src/game/constants.rs`

盤面サイズやマス数など、ゲーム全体で使う定数を置きます。

現在の主な定数:

- `BOARD_SIZE`: 盤面の一辺の長さ
- `CELL_COUNT`: 盤面全体のマス数
- `COLUMN_COUNT`: コマを落とせる柱の本数

理由:

- `4` や `64` のような数字をコード中に散らばらせない
- 盤面サイズに関する前提を一箇所で読める
- 将来、小さい盤面で実験したくなったときに変更点を見つけやすい

### `src/game/cell.rs`

盤面の1マスを表す `Cell` を置きます。

`Cell` は、マスが空か、白のコマか、黒のコマかを表します。

理由:

- 盤面の中身だけに関する型なので、独立させると読みやすい
- 将来、数値化するときに `Cell` から `0, 1, 2` へ変換する処理を足しやすい

### `src/game/player.rs`

手番を表す `Player` を置きます。

`Player` は「これから打つ人」を表します。`Cell` と似ていますが、`Cell` は盤面上の状態、`Player` は行動する主体です。

理由:

- `Player::cell()` のように、プレイヤーから置くコマへ変換する処理をまとめられる
- `Player::next()` のように、手番交代の処理をまとめられる
- `Cell` と `Player` を分ける理由がファイル単位でも見えやすい

### `src/game/coordinate.rs`

座標を表す型を置きます。

現在の型:

- `Column`: プレイヤーが選ぶ柱 `(x, y)`
- `Position`: 実際にコマが置かれた3次元座標 `(x, y, z)`

理由:

- 重力ありルールでは「選ぶ場所」と「置かれる場所」が違う
- 勝敗判定では `Position` が重要になる
- 座標変換や対称性の実装を将来ここから発展させやすい

### `src/game/line.rs`

勝敗判定で使う方向を置きます。

現在の主な型と定数:

- `Direction`: 3次元盤面上で1マス進む向き
- `ALL_DIRECTIONS`: 勝敗判定で調べる代表13方向

理由:

- 最後に置かれた `Position` から、どの向きに同じ色が並ぶかを調べるため
- 正方向と逆方向をセットで調べれば、26方向ではなく13方向で済むため
- 勝敗判定の前段階として、方向の考え方を独立して学べるため

### `src/game/state.rs`

ゲーム状態と、状態を変化させる関数を置きます。

現在の主な型と関数:

- `Board`: 3次元配列の盤面
- `GameState`: 盤面、手番、手数をまとめた状態
- `PlayResult`: 着手後の状態と、最後に置かれた位置
- `PlayError`: 着手できなかった理由
- `GameState::initial()`: 初期状態を作る
- `GameState::legal_moves()`: 合法手を列挙する
- `GameState::play()`: 1手打って次状態を作る

理由:

- 状態遷移はゲーム解析の中心なので、独立したファイルにする
- テストもここに置くことで、状態定義と期待する動作を近い場所で読める
- 将来、勝敗判定が増えたら `state.rs` に直接足すか、別ファイルへ切り出すか判断しやすい

## 今後増やす可能性があるファイル

### `src/game/status.rs`

ゲームが進行中か、勝ちか、引き分けかを表す型を置く候補です。

例えば `GameStatus::InProgress`、`GameStatus::Win(Player)`、`GameStatus::Draw` のような型を作る段階で検討します。

### `src/solver/`

完全解析やミニマックス探索を置く候補です。

`game` はルールそのもの、`solver` はそのルールを使って勝敗を解く処理、という分け方にします。

### `src/storage/`

状態や解析結果をファイルに保存する処理を置く候補です。

テキスト形式、JSON、バイナリ形式などを試す段階で検討します。

## 分割の考え方

最初から細かく分けすぎると、学習時にファイルを行き来する負担が増えます。一方で、すべてを `main.rs` に置くと、型が増えたときに全体像が見えにくくなります。

このリポジトリでは、DDDやクリーンアーキテクチャを厳密に実践することは目的にしません。ただし、「ドメインを中心に置く」という考え方は意識します。

ここでいうドメインは、立体4目並べのルールそのものです。例えば、盤面、手番、合法手、重力による落下、勝敗判定はドメインに含まれます。一方で、探索結果をファイルへ保存することや、コマンドラインで表示することは、ゲームルールそのものではありません。

このリポジトリでは、次の方針で分けます。

- 実行の入口は `main.rs`
- 再利用したいロジックは `lib.rs` から公開する
- ゲームのルールや状態表現は `game` に置く
- 探索は `solver`、保存は `storage` のように、役割が明確になってから増やす
- 学習のため、ファイルを分けた理由をこの文書に残す

依存方向もなるべく単純にします。

```text
main.rs
  ├── game
  ├── solver
  └── storage

solver  -> game
storage -> game
game    -> できるだけ他の層に依存しない
```

`game` はルールの中心なので、保存形式や探索アルゴリズムの都合をなるべく知らない形にします。例えば、`GameState::play` はゲームルールなので `game` に置きます。一方で、ミニマックス探索やメモ化はルールを使う側なので、将来 `solver` に置く方針です。
