# ADR-0005: 探索結果のメモ化を `MemoTable` で始める

- Status: Accepted
- Date: 2026-05-04

## 背景

完全解析では、同じ局面を何度も評価すると探索量が増える。
そのため、一度調べた局面の結果を保存し、同じ局面に再び到達したら保存済みの結果を取り出す必要がある。

この考え方をメモ化と呼ぶ。

現時点では、まだミニマックス探索本体は実装しない。
まずは「局面をキーにする」「結果を保存する」「同じ局面から結果を取り出す」という最小単位を理解する。

## 決定

- 探索用のコードは `src/solver/` に置く。
- 探索結果を `Outcome` として定義する。
  - `Win`
  - `Loss`
  - `Draw`
- `Outcome` は「次に手を打つプレイヤーから見た結果」として読む。
- メモ化用に `MemoTable` を定義する。
- `MemoTable` は内部で `HashMap<u128, Outcome>` を持つ。
- キーには `GameState::board_key_base3()` を使う。

## 理由

- `game` は立体4目並べのルールそのものを表す層として保ちたい。探索やメモ化はルールの外側でゲームを解く処理なので、`solver` に分ける。
- `HashMap<u128, Outcome>` を直接あちこちで使うより、`MemoTable` に包む方が「何をキーにして何を保存するのか」が読みやすい。
- `Outcome` を `GameStatus` と分けることで、「ゲームの進行状態」と「探索で見た勝ち負け」を混ぜずに済む。

## 影響

- 現在の `MemoTable` はメモリ上の保存だけを扱う。ファイル保存はまだ扱わない。
- 対称性を考慮した正規化キーはまだ使わない。必要になったら `MemoTable` が使うキー生成方法を見直す。
- `Outcome` の意味は「手番側から見た結果」なので、終局状態の `GameStatus::Win(Player)` から変換するときは、勝者と `state.turn` の関係に注意する必要がある。
