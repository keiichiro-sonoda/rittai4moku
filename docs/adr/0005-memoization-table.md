# ADR-0005: 探索結果のメモ化を `MemoTable` で始める

- Status: Accepted
- Date: 2026-05-04

## 背景

完全解析では、同じ局面を何度も評価すると探索量が増える。
そのため、一度調べた局面の結果を保存し、同じ局面に再び到達したら保存済みの結果を取り出す必要がある。

この考え方をメモ化と呼ぶ。

現時点では、まだミニマックス探索本体は実装しない。
まずは「局面をキーにする」「結果を保存する」「同じ局面から結果を取り出す」という最小単位を理解する。

## 決定

- 探索用のコードは `src/solver/` に置く。
- 探索結果を `Outcome` として定義する。
  - `Win`
  - `Loss`
  - `Draw`
- 現時点の `Outcome` には、勝ち負けまでの手数は持たせない。
- `Outcome` は「次に手を打つプレイヤーから見た結果」として読む。
- `Outcome::flip()` で、相手から見た結果を自分から見た結果へ反転する。
- `Outcome::from_status_for_player(status, perspective)` で、終局状態の `GameStatus` を指定プレイヤー視点の `Outcome` に変換する。
- メモ化用に `MemoTable` を定義する。
- `MemoTable` は内部で `HashMap<u128, Outcome>` を持つ。
- キーには `GameState::board_key_base3()` を使う。

## 理由

- `game` は立体4目並べのルールそのものを表す層として保ちたい。探索やメモ化はルールの外側でゲームを解く処理なので、`solver` に分ける。
- `HashMap<u128, Outcome>` を直接あちこちで使うより、`MemoTable` に包む方が「何をキーにして何を保存するのか」が読みやすい。
- `Outcome` を `GameStatus` と分けることで、「ゲームの進行状態」と「探索で見た勝ち負け」を混ぜずに済む。
- `GameStatus::InProgress` はまだ結果ではないため、`Outcome` への変換結果は `None` にする。進行中の局面は、合法手を調べて初めて `Outcome` が決まる。
- 再帰探索では、自分が1手打った後の子局面は相手番になる。子局面の結果は相手視点なので、親局面へ戻すときに `flip()` が必要になる。
- 完全解析の最初の目標は、すべての到達可能局面を `Win` / `Loss` / `Draw` のいずれかに分類すること。葉からの手数は、分類そのものには必須ではない。

## 影響

- 現在の `MemoTable` はメモリ上の保存だけを扱う。ファイル保存はまだ扱わない。
- 対称性を考慮した正規化キーはまだ使わない。必要になったら `MemoTable` が使うキー生成方法を見直す。
- `Outcome` の意味は「手番側から見た結果」なので、終局状態の `GameStatus::Win(Player)` から変換するときは、勝者と `state.turn` の関係に注意する必要がある。
- 対戦AIとして「できるだけ早く勝つ」「できるだけ遅く負ける」手を選びたい場合は、将来 `Outcome::Win { plies }` / `Outcome::Loss { plies }` のように、葉までの半手数を持つ表現へ拡張する。その変更は別 ADR で扱う。

## 将来の `plies` の定義

`plies` は半手数を表す。ここでいう半手とは、どちらか一方のプレイヤーが1回コマを落とすこと。
このリポジトリの `moves_played` と同じ数え方で、黒の着手で1、白の着手でさらに1増える。

将来 `plies` を `Outcome` に持たせる場合、意味は次のように定義する。

- `Win { plies: n }`: 現在の手番側が、現在局面から `n` 回の着手後に勝ちを成立させる。
- `Loss { plies: n }`: 現在の手番側が、現在局面から `n` 回の着手後に相手に勝ちを成立させられる。
- `Draw`: 勝敗が成立しないので、勝ち負けまでの `plies` は持たない。

この定義では、通常 `Win` の `plies` は奇数、`Loss` の `plies` は偶数になる。
自分の勝ちは自分の着手で成立するため、`1, 3, 5, ...` 半手後になる。
自分の負けは相手の着手で成立するため、`2, 4, 6, ...` 半手後になる。
