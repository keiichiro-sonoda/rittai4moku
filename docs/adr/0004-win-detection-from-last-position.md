# ADR-0004: 勝敗判定は最後に置かれた `Position` 起点で代表13方向のみ調べる

- Status: Accepted
- Date: 2026-05-04

## 背景

3 次元盤面では、各軸に `-1, 0, 1` の進み方があり、`(0, 0, 0)` を除くと 26 方向。ただし、ある直線は正方向と逆方向で同じ直線を指すため、代表は半分の 13 方向で足りる。

選択肢は次のとおり。

- 盤面全体の全勝ちラインを毎回走査する
- 最後に置かれたコマを起点に、そこを通る直線だけ走査する

## 決定

- `ALL_DIRECTIONS` を 13 方向の代表として保持する。内訳は軸方向 3 + 平面斜め 6 + 空間斜め 4。
- 勝敗判定は `is_winning_position(start)` を `start = placed_at`（最後に置かれた位置）で呼ぶ。13 方向それぞれで `count_line_cells(start, dir, target) >= BOARD_SIZE` を確認する。
- `count_line_cells` は「`start` 自身 + 正方向に続く同色数 + 逆方向に続く同色数」を返す。
- `status_after_move(placed_at)` の判定順は次のとおり。
  1. 勝ちが成立していれば `Win(player)`
  2. 勝ちが無く盤面が満杯なら `Draw`
  3. それ以外は `InProgress`

## 理由

- 勝ちは最後の手によってのみ新しく成立する。前の手で成立済みなら、その時点で検出済みになっている前提に立つ。
- 1 手あたりに調べるラインは 13 方向 × 最大 4 マス程度で済み、盤面全勝ちラインを毎回走査するより明確に少ない。
- 起点を中央に置けば左右両方を数える必要があるが、`count_line_cells` で 1 関数にまとめられるためコード分岐が単純になる。

## 影響

- `play` を経由せず `board` を直接組み立てるテストでも勝敗判定が成立する必要がある。既存テスト `is_winning_position_detects_all_representative_directions` がこれを担保している。
- 盤面全体を走査する勝敗判定は持たない。デバッグ用途で必要になった時点で追加し、本 ADR を更新する。
- 「前の手までに勝ちが残っていない」という不変条件に依存している。`GameState` を外から組み立てる API（例: `from_board_key_base3`）の戻り値に対して `status_after_move` を呼ぶ場合、呼び出し側がこの不変条件を保証する。
