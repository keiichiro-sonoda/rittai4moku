# ADR-0001: 盤面表現を `[[[Cell; 4]; 4]; 4]` (`board[z][y][x]`) にする

- Status: Accepted
- Date: 2026-05-04

## 背景

立体4目並べの盤面は 4×4×4 = 64 マスで、各マスの状態は空・黒・白の 3 値。表現の候補は次のあたりが現実的だった。

- ビット盤（黒用と白用の `u64` をペアで持つ）
- 1 次元配列 `[Cell; 64]`（座標は `z * 16 + y * 4 + x` のように計算）
- 3 次元配列 `[[[Cell; 4]; 4]; 4]`
- `Vec<Cell>` 系

## 決定

盤面型を `Board = [[[Cell; BOARD_SIZE]; BOARD_SIZE]; BOARD_SIZE]` とし、添字は `board[z][y][x]` で読む。

- `x`: 横方向
- `y`: 奥行き方向
- `z`: 高さ。`0` が最下段、`3` が最上段。

`Cell` は `enum { Empty, Black, White }` のままで持つ。

## 理由

- 学習段階では「何のマスを見ているのか」が型と添字から読み取れることを最優先する。3 次元配列は座標との対応が直接で、ビット盤よりコードを追いやすい。
- `z` を最も外側に置くと、ある柱 `(x, y)` を下から走査する処理を `(0..BOARD_SIZE).find(|&z| board[z][y][x] == Empty)` のように直感的に書ける。
- `Cell` を enum のまま持つことでパターンマッチが効き、`Empty` / `Black` / `White` の取り違えがコンパイル時に気付ける。

## 影響

- メモリ効率は最適ではない（1 マスにつき 1 バイト + アラインメント）。完全解析や置換表で問題になった段階でビット盤への変換関数を別途用意する。本 ADR を Superseded にする必要はないが、ビット盤側の表現は別 ADR で記録する。
- 添字順 `[z][y][x]` を覚える必要がある。AGENTS.md と `Board` のドックコメントに明記してある。
