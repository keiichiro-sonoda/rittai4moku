# ADR-0003: 盤面キーを 3進数 (`u128`, z→y→x 走査) で表す

- Status: Accepted
- Date: 2026-05-04

## 背景

探索結果（最善手・評価値・到達フラグ）を `HashMap` などに保存するためには、盤面を一意なキーに変換する必要がある。マス数 64、状態数 3 という構造から、3 進数表現がそのまま当てはまる。

`3^64 ≈ 3.4 × 10^30` は `2^128 ≈ 3.4 × 10^38` の範囲内に収まるため、`u128` 1 つで盤面全体を表せる。

## 決定

- `board_key_base3` は盤面を 3 進数の `u128` に符号化する。
- 各マスの桁: `Empty = 0`, `Black = 1`, `White = 2`。
- 走査順は外側から `z → y → x`。`(0, 0, 0)` を最下位桁、`(3, 3, 3)` を最上位桁とする。
- 逆変換 `from_board_key_base3` は同じ走査順で `board` を組み立て、非空マスの個数から `moves_played` を、その偶奇から `turn` を復元する。
- `3^64` 以上の値は 64 マスに収まらないため、キーとして不正とみなして `None` を返す。

## 理由

- 3 進数はマス状態を無駄なく表現できる。64 桁ぴったりが `u128` に収まる。
- 走査順を仕様として固定することで、キーの再現性とテスト可能性を担保する。
- 黒を `1`、白を `2` にしたのは「先手の黒を先に扱う」と読みやすいため。

## 影響

- 復元時には到達可能性まで検証しない。重力違反や黒白枚数差は別関数の責務になる。必要になった時点で別 ADR を立てる。
- ビット盤に切り替える場合は、本 ADR を Superseded にして新しいキー仕様を別 ADR で作る。3 進キーとビット盤キーを併存させる場合も同様に別 ADR で扱う。
- 盤面の対称性（回転・鏡映）を吸収した正規化キーを導入する場合、本 ADR の上に追加 ADR を重ねる。本 ADR 自体は対称性を扱わない。
