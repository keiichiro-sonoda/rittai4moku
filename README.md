# rittai4moku

立体4目並べ（4×4×4 の重力あり版）を題材に、Rust とゲーム解析の考え方を学ぶための実験リポジトリ。

「最短で強いプログラムを完成させる」のが目的ではなく、二人零和有限確定完全情報ゲームをどのように状態として表し、どのように探索・評価・保存していくのかを、コードを書きながら理解することを意図している。コード本体のドックコメントは、そのまま学習ノートとして読める粒度で書いてある。

## ルール

- 盤面は 4×4×4 の立方体。プレイヤーは上から穴の開いたコマを柱に通して落とす。
- 同じ柱では一番下から順番にコマが積み上がる（重力あり）。
- 縦・横・奥行・各種斜めのいずれかで自分のコマを 4 つ並べたら勝ち。
- 先手は黒、後手は白。

## 動かし方

```bash
cargo build           # ビルド
cargo test            # 単体テストと統合テストを実行
cargo run             # 現状は初期状態を表示するだけ
```

整形は `cargo fmt --all`、静的解析は `cargo clippy --all-targets -- -D warnings`。

## 設計の考え方（要点）

ここでは要約だけ並べる。各判断の背景と理由は `docs/adr/` に 1 ファイルずつ書いてある。

- **状態と評価ラベルを分ける**。`GameState` は盤面・手番・手数を持つデータ本体、`GameStatus` はそこから得られる「進行中・勝ち・引き分け」のラベル。
- **プレイヤーが選ぶのは柱、実際の置き場所は重力で決まる**。前者を `Column`、後者を `Position` として型で区別する。
- **勝敗判定は最後に置かれた位置から代表13方向だけ**を調べる（[ADR-0004](docs/adr/0004-win-detection-from-last-position.md)）。
- **盤面は 3進数キー（`u128`）に往復できる**（[ADR-0003](docs/adr/0003-base3-board-key.md)）。探索結果を `HashMap` などに保存する将来計画の足場。
- **盤面表現は速度より読みやすさを優先する**（[ADR-0001](docs/adr/0001-board-representation.md)、[ADR-0002](docs/adr/0002-redundant-turn-and-moves-played.md)）。

## 学習リポジトリとしてのスタンス

- 速度より読みやすさを優先する。最適化はベンチマークで裏付けてから入れる。
- 盤面から復元できる情報（`turn`, `moves_played`）も、学習段階では明示的に状態へ持つ。理由が不要になったら ADR を更新したうえで削る。
- ドックコメントには「なぜそうしているか」を残す。コード自体が学習ノートになるように書く。

- 設計判断の根拠 → `docs/adr/`
