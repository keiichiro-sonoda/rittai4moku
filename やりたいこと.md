# やりたいこと

## このリポジトリの位置づけ

このリポジトリは、立体4目並べを題材にした学習用リポジトリです。

目的は、単に完成した解析プログラムを作ることではありません。Rust でコードを書きながら、次の2つを少しずつ理解することを重視します。

- Rust の基本的な型、配列、構造体、列挙型、所有権、テスト、ファイル入出力を学ぶ
- 二人零和有限確定完全情報ゲームを、状態・合法手・終局判定・探索・保存・正規化として扱う方法を学ぶ

完成度よりも、各段階で「なぜその表現にするのか」「何が分かるようになったのか」を記録することを優先します。

コードの置き場所やモジュール分割の方針は、[ディレクトリ構造.md](ディレクトリ構造.md) にまとめます。実装が増えてきたら、この文書も一緒に更新します。

設計では、DDDやクリーンアーキテクチャを厳密に導入するのではなく、その考え方を軽く意識します。具体的には、立体4目並べのルールそのものを中心に置き、探索・保存・表示の都合をゲームルールに混ぜないようにします。

型やモジュールの関係は、[docs/uml.md](docs/uml.md) に Mermaid 図として残します。Rust のコードと図を見比べながら、構造を理解できるようにします。

## 対象ゲーム

立体4目並べを `4 x 4 x 4` の盤面として扱います。

- プレイヤーは黒と白の2人
- 先手は黒とする
- 4 x 4 本の柱がある
- 1手につき、プレイヤーは柱の座標 `(x, y)` を1つ選ぶ
- コマは上から落ち、同じ柱の一番低い空き高さ `z` に入る
- `z = 0` が一番下、`z = 3` が一番上
- 同じ `(x, y)` の柱で `z = 0` が空なのに `z = 1` に置くことはできない
- `z = 0, 1, 2, 3` がすべて埋まった柱は合法手ではない
- まだどの柱も満杯でなければ、合法手は先手・後手ともに16通り
- 縦・横・奥行き・斜め方向のいずれかで4つ並べば勝ち
- すべてのマスが埋まり、どちらも勝っていなければ引き分け

このリポジトリでは、実物の立体4目並べに合わせて「重力あり」のルールを採用します。つまり、プレイヤーが直接選ぶのは3次元座標 `(x, y, z)` ではなく、上からコマを落とす柱 `(x, y)` です。

## 今回のルール整理で変えた考え方

最初のメモでは「空いている3次元のマスを選ぶ」ような状態表現を考えていました。しかし、実物の立体4目並べには重力があります。そのため、コード上でも次のように考え方を変えます。

- 盤面 `Board` は引き続き `board[z][y][x]` の3次元配列で持つ
- ただし、合法手は `Position { x, y, z }` ではなく `Column { x, y }` として表す
- `z` はプレイヤーが選ぶ値ではなく、現在の盤面から自動的に決まる値にする
- `GameState::next_empty_z` で、指定された柱の次に入る高さを調べる
- `GameState::legal_moves` では、満杯ではない柱だけを合法手として返す
- `GameState::play` では、柱にコマを落として次の状態と実際に置かれた位置を返す

この変更により、「ルール上ありえない状態」を作りにくくなります。例えば `z = 0` が空なのに `z = 1` にだけコマがある状態は、`play` を使っている限り発生しません。

## 最後に置かれた位置を覚える理由

勝敗判定では、毎回すべての勝利ラインを調べる方法もあります。しかし、直前の手で勝ちが発生したかどうかを知りたいなら、最後に置かれたコマを通るラインだけを調べれば十分です。

そのため、コードでは次の2つを分けます。

- `Column`: プレイヤーが選ぶ柱 `(x, y)`
- `Position`: 実際にコマが置かれた3次元座標 `(x, y, z)`

重力ありルールでは、プレイヤーは `z` を選びません。例えば `Column { x: 2, y: 1 }` を選んだとき、空の柱なら `Position { x: 2, y: 1, z: 0 }` に置かれます。同じ柱にもう一度置くと、次は `Position { x: 2, y: 1, z: 1 }` になります。

`GameState::play` は `PlayResult` を返します。`PlayResult` には、着手後の `state` と、最後に置かれた `placed_at` が入っています。これにより、次の段階で `placed_at` を入口にして勝敗判定を作れます。

## 勝敗判定の最適化方針

`GameState::is_winning_position` では、最初に起点のマスを1回だけ確認します。起点が `Cell::Empty` なら、その時点で勝ちはありえないので `false` を返します。これにより、空きマスに対して13方向すべてで同じ空チェックを繰り返すことを避けます。

一方で、「この位置と方向では4つ並ぶ可能性がない」という方向を事前に省く最適化は、今は実装しません。例えば空間斜めは端に近い位置でしか4マス分伸びないため、省ける方向があります。しかし、この判定を入れると、座標と方向の組み合わせを正しく扱う追加ロジックが必要になり、勝敗判定そのものの理解が難しくなります。

今の方針:

- 起点が空かどうかのような、簡単で読みやすくなる改善は入れる
- 位置ごとに不要な方向を省く最適化は後回しにする
- まず正しい勝敗判定を完成させる
- 完全解析に進んで性能を測り、必要になったら `can_fit_four(position, direction)` のような関数を検討する

## 命名と冗長な状態情報について

`GameState` と `GameStatus` は名前が似ていますが、このリポジトリでは次のように使い分けます。

- `GameState`: ゲームを進めるために必要なデータ本体
- `GameStatus`: その状態を評価した結果、ゲームが進行中・勝ち・引き分けのどれかを表すラベル

英語では、`state` は内部状態や局面、`status` は進行状況や評価結果という意味で使われることが多いため、この命名を採用します。

また、`GameState` は `turn` と `moves_played` を持っています。現在のルールでは、先手が黒で、白黒が必ず交互に打ち、コマが盤面から消えないため、この2つは盤面から復元できます。

それでも学習段階では、次の理由で明示的に持ちます。

- `play` の処理が読みやすい
- `is_full` が単純になる
- デバッグ時に、手番と手数をすぐ確認できる
- ゲーム解析で「状態には手番も含める」という考え方を明示できる

一方で、冗長情報には `board` と矛盾する可能性があります。将来、状態の数値化や完全解析へ進む段階で、この冗長さが問題になったら、`board` だけを真実として `turn()` や `moves_played()` を計算する設計に見直す可能性があります。

盤面にコマを置くためには、柱と手を表す型が必要です。

理解したいこと:

- Rust でエラーを `Option` や `Result` として表す方法
- 重力によって `z` が自動的に決まることを、コードでどう表すか

実装タスク:

- `Column` を定義する
  - `x`
  - `y`
- 必要になったら、実際にコマが入った場所を表す `Position` を定義する
  - `x`
  - `y`
  - `z`
- `PlayResult` を定義する
  - `state`: 着手後の状態
  - `placed_at`: 実際にコマが置かれた3次元座標
- 柱が盤面内か確認する関数を作る
- 指定した柱の次に空いている高さ `z` を返す関数を作る
- 柱が満杯か確認する関数を作る
- 1手進めた次状態を返す関数を作る
  - 元の状態を変更する版
  - 新しい状態を返す版
  - どちらが学習しやすいか比較する

確認したい問い:

- `GameState` をコピーして次状態を作るのは簡単だが、探索では遅くなるか
- Rust の借用を使って盤面を直接更新すると、コードは分かりやすいか
- 範囲外の柱や満杯の柱を `panic!` にするべきか、`Result` で返すべきか
- `z` を外から受け取らないことで、どんな不正状態を防げるか
- `play` が状態だけを返す場合と、最後に置かれた位置も返す場合で、次の勝敗判定の作りやすさはどう変わるか

  - `Draw`

  - `dy`
  - `dz`

- 完全解析ではどちらの判定方法が扱いやすいか
- 重力は合法手に影響するが、勝敗判定の直線定義にはどう影響するか
- `Position` があると、どの方向を調べればよいかをどう表現できるか
- 26方向ではなく13方向だけ定義すればよい理由は何か

## 進め方の原則

- 1回に理解する概念を増やしすぎない
- 実装したら、小さなテストを必ず書く
- なぜその型にしたのかをコメントかメモに残す
- 速さよりも、まず正しさと説明しやすさを優先する
- 完全解析に必要な最適化は、正しい基準実装ができてから検討する
