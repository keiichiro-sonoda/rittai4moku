# やりたいこと

## このリポジトリの位置づけ

このリポジトリは、立体4目並べを題材にした学習用リポジトリです。

目的は、単に完成した解析プログラムを作ることではありません。Rust でコードを書きながら、次の2つを少しずつ理解することを重視します。

- Rust の基本的な型、配列、構造体、列挙型、所有権、テスト、ファイル入出力を学ぶ
- 二人零和有限確定完全情報ゲームを、状態・合法手・終局判定・探索・保存・正規化として扱う方法を学ぶ

完成度よりも、各段階で「なぜその表現にするのか」「何が分かるようになったのか」を記録することを優先します。

コードの置き場所やモジュール分割の方針は、[ディレクトリ構造.md](ディレクトリ構造.md) にまとめます。実装が増えてきたら、この文書も一緒に更新します。

設計では、DDDやクリーンアーキテクチャを厳密に導入するのではなく、その考え方を軽く意識します。具体的には、立体4目並べのルールそのものを中心に置き、探索・保存・表示の都合をゲームルールに混ぜないようにします。

## 対象ゲーム

立体4目並べを `4 x 4 x 4` の盤面として扱います。

- プレイヤーは黒と白の2人
- 先手は黒とする
- 4 x 4 本の柱がある
- 1手につき、プレイヤーは柱の座標 `(x, y)` を1つ選ぶ
- コマは上から落ち、同じ柱の一番低い空き高さ `z` に入る
- `z = 0` が一番下、`z = 3` が一番上
- 同じ `(x, y)` の柱で `z = 0` が空なのに `z = 1` に置くことはできない
- `z = 0, 1, 2, 3` がすべて埋まった柱は合法手ではない
- まだどの柱も満杯でなければ、合法手は先手・後手ともに16通り
- 縦・横・奥行き・斜め方向のいずれかで4つ並べば勝ち
- すべてのマスが埋まり、どちらも勝っていなければ引き分け

このリポジトリでは、実物の立体4目並べに合わせて「重力あり」のルールを採用します。つまり、プレイヤーが直接選ぶのは3次元座標 `(x, y, z)` ではなく、上からコマを落とす柱 `(x, y)` です。

## 今回のルール整理で変えた考え方

最初のメモでは「空いている3次元のマスを選ぶ」ような状態表現を考えていました。しかし、実物の立体4目並べには重力があります。そのため、コード上でも次のように考え方を変えます。

- 盤面 `Board` は引き続き `board[z][y][x]` の3次元配列で持つ
- ただし、合法手は `Position { x, y, z }` ではなく `Column { x, y }` として表す
- `z` はプレイヤーが選ぶ値ではなく、現在の盤面から自動的に決まる値にする
- `GameState::next_empty_z` で、指定された柱の次に入る高さを調べる
- `GameState::legal_moves` では、満杯ではない柱だけを合法手として返す
- `GameState::play` では、柱にコマを落として次の状態と実際に置かれた位置を返す

この変更により、「ルール上ありえない状態」を作りにくくなります。例えば `z = 0` が空なのに `z = 1` にだけコマがある状態は、`play` を使っている限り発生しません。

## 最後に置かれた位置を覚える理由

勝敗判定では、毎回すべての勝利ラインを調べる方法もあります。しかし、直前の手で勝ちが発生したかどうかを知りたいなら、最後に置かれたコマを通るラインだけを調べれば十分です。

そのため、コードでは次の2つを分けます。

- `Column`: プレイヤーが選ぶ柱 `(x, y)`
- `Position`: 実際にコマが置かれた3次元座標 `(x, y, z)`

重力ありルールでは、プレイヤーは `z` を選びません。例えば `Column { x: 2, y: 1 }` を選んだとき、空の柱なら `Position { x: 2, y: 1, z: 0 }` に置かれます。同じ柱にもう一度置くと、次は `Position { x: 2, y: 1, z: 1 }` になります。

`GameState::play` は `PlayResult` を返します。`PlayResult` には、着手後の `state` と、最後に置かれた `placed_at` が入っています。これにより、次の段階で `placed_at` を入口にして勝敗判定を作れます。

## 学習フェーズ

### 1. 状態を定義する

まず、ゲームを解析するための最小単位として「状態」を定義します。

理解したいこと:

- ゲーム状態とは何か
- 盤面だけでなく、手番も状態に含める必要がある理由
- Rust の `enum` と `struct` をどう使い分けるか
- `const` と `const fn` の違い
- 3次元配列で盤面を表すと何が分かりやすく、何が扱いづらいか

実装タスク:

- `Cell` を定義する
  - `Empty`
  - `White`
  - `Black`
- `Player` を定義する
  - `White`
  - `Black`
- `Board` を `[[[Cell; 4]; 4]; 4]` として定義する
- `GameState` を定義する
  - `board`
  - `turn`
  - `moves_played`
- 初期状態を作る
  - `GameState::initial()`
  - `INITIAL_STATE`
- 初期状態のテストを書く

確認したい問い:

- 同じ盤面でも、次が黒番か白番かで結果は変わるか
- `moves_played` は盤面から数えられるのに、状態に持つべきか
- `Cell` と `Player` を同じ型にしない理由は何か

### 2. 座標と着手を定義する

盤面にコマを置くためには、柱と手を表す型が必要です。

理解したいこと:

- 3次元座標 `(x, y, z)` と、プレイヤーが実際に選ぶ柱 `(x, y)` の違い
- 柱をタプルで済ませる場合と、構造体にする場合の違い
- 範囲外の柱や、すでに満杯の柱をどう扱うか
- Rust でエラーを `Option` や `Result` として表す方法
- 重力によって `z` が自動的に決まることを、コードでどう表すか

実装タスク:

- `Column` を定義する
  - `x`
  - `y`
- 必要になったら、実際にコマが入った場所を表す `Position` を定義する
  - `x`
  - `y`
  - `z`
- `PlayResult` を定義する
  - `state`: 着手後の状態
  - `placed_at`: 実際にコマが置かれた3次元座標
- 柱が盤面内か確認する関数を作る
- 指定した柱の次に空いている高さ `z` を返す関数を作る
- 柱が満杯か確認する関数を作る
- 1手進めた次状態を返す関数を作る
  - 元の状態を変更する版
  - 新しい状態を返す版
  - どちらが学習しやすいか比較する

確認したい問い:

- `GameState` をコピーして次状態を作るのは簡単だが、探索では遅くなるか
- Rust の借用を使って盤面を直接更新すると、コードは分かりやすいか
- 範囲外の柱や満杯の柱を `panic!` にするべきか、`Result` で返すべきか
- `z` を外から受け取らないことで、どんな不正状態を防げるか
- `play` が状態だけを返す場合と、最後に置かれた位置も返す場合で、次の勝敗判定の作りやすさはどう変わるか

### 3. 合法手を列挙する

ある状態から次に打てる手をすべて列挙します。

理解したいこと:

- ゲーム木とは何か
- 分岐数とは何か
- 合法手生成が間違っていると、完全解析全体が間違う理由
- イテレータと `Vec` の使い分け

実装タスク:

- 4 x 4 の柱を走査する
- 満杯ではない柱だけを合法手として集める
- `legal_moves(&self) -> Vec<Column>` のような関数を作る
- 初期状態では16手あることをテストする
- 1手打っただけなら、同じ柱にまだ置けるので合法手は16手のままであることを確認する
- 同じ柱に4手置いた後は、その柱が合法手から消えて15手になることをテストする

確認したい問い:

- 合法手を毎回 `Vec` に集める方法は分かりやすいか
- 後で高速化するとしたら、どの部分がボトルネックになりそうか
- 重力ありルールでは、状態数や分岐数は重力なしルールと比べてどう変わるか

### 4. 勝敗判定を実装する

4つ並んでいるかを判定します。

理解したいこと:

- 終局状態とは何か
- 勝ち、負け、引き分け、未終了をどう型で表すか
- 立体盤面の直線をどう列挙するか
- 判定ロジックをテストで支える方法

実装タスク:

- `GameResult` または `GameStatus` を定義する
  - `InProgress`
  - `Win(Player)`
  - `Draw`
- まずは最後に置かれた `Position` を通る方向だけを調べる方法を検討する
- 必要なら、比較用に4マスからなるすべての勝利ラインを列挙する
- 指定プレイヤーが勝っているかを判定する
- 状態全体の終了判定を作る
- 重力ありでも勝利ラインの種類自体は3次元方向に存在することを確認する
- 軸方向の勝ちをテストする
- 平面斜めの勝ちをテストする
- 空間斜めの勝ちをテストする

確認したい問い:

- 勝利ラインは手で列挙するべきか、プログラムで生成するべきか
- 最後に置いた手だけを見れば勝敗判定できるか
- 完全解析ではどちらの判定方法が扱いやすいか
- 重力は合法手に影響するが、勝敗判定の直線定義にはどう影響するか
- `Position` があると、どの方向を調べればよいかをどう表現できるか

### 5. 状態を数値化する

探索結果を保存したり、同じ状態を再利用したりするため、状態をキーに変換します。

理解したいこと:

- 状態空間とは何か
- ハッシュ化と一意な数値化の違い
- `HashMap` を使う意味
- 盤面を3進数として読む考え方

実装タスク:

- `Cell` を `0, 1, 2` に変換する関数を作る
- 盤面を1次元順に走査する順番を決める
- 盤面を3進数のように数値化する
- 手番もキーに含める
- 数値化した状態を元に戻す関数を検討する

確認したい問い:

- 状態キーに `moves_played` は必要か
- 数値化が衝突しないとはどういう意味か
- `u128` に収まるか、別の表現が必要か

### 6. 探索で勝敗を解く

ある状態が「勝ち」「負け」「引き分け」のどれになるかを再帰的に調べます。

理解したいこと:

- ミニマックス法とは何か
- 現在の手番から見た勝ち・負けとは何か
- 後退解析と前向き探索の違い
- メモ化とは何か
- 「勝てる手が1つでもあれば勝ち」「すべて負けに向かうなら負け」という考え方

実装タスク:

- `Outcome` を定義する
  - `Win`
  - `Loss`
  - `Draw`
- 終局状態を `Outcome` に変換する
- 再帰的に合法手を試す
- 勝てる手を見つけたらそこで探索を打ち切る
- `HashMap` で探索済み状態を保存する
- 小さい盤面サイズで動作確認できるようにすることを検討する

確認したい問い:

- 勝ちを見つけた時点で他の手を調べなくてよい理由は何か
- 「相手にとっての負け」は「自分にとっての勝ち」と言えるか
- 再帰が深くなりすぎる問題をどう考えるか

### 7. 対称性を使って状態を正規化する

回転や鏡写しで同じとみなせる盤面を、同じ状態として扱います。

理解したいこと:

- ゲームの対称性とは何か
- 正規化とは何か
- 状態数を減らすことが探索にどう効くか
- 盤面変換を間違えると何が壊れるか

実装タスク:

- 立方体の回転・鏡写しをどこまで同一視するか決める
- 座標変換関数を作る
- 変換後の盤面を生成する
- すべての同値状態を数値化する
- 最小の数値キーを正規形として採用する
- 正規化前後で勝敗が変わらないことをテストする

確認したい問い:

- 手番は正規化で変わるか
- 黒白を入れ替える対称性は使ってよいか
- ルールに重力がある場合、上下をひっくり返す変換は使ってよいか
- 重力方向を変えてしまう対称変換は、合法状態を合法状態に保てるか

### 8. 盤面データを保存する

探索結果や途中状態をファイルに保存します。

理解したいこと:

実装タスク:

確認したい問い:

## 進め方の原則

- 1回に理解する概念を増やしすぎない
- 実装したら、小さなテストを必ず書く
- なぜその型にしたのかをコメントかメモに残す
- 速さよりも、まず正しさと説明しやすさを優先する
- 完全解析に必要な最適化は、正しい基準実装ができてから検討する

## 現在の次の一歩

現在は「状態を定義する」段階です。

重力ありルールを反映し、次の要素をコードに入れました。

- `Column`: プレイヤーが選ぶ柱
- `Position`: 実際にコマが置かれた3次元座標
- `PlayResult`: 着手後の状態と、最後に置かれた位置のまとまり
- `PlayError`: 着手できなかった理由
- `GameState::next_empty_z`: 指定した柱の次に入る高さを調べる関数
- `GameState::legal_moves`: 満杯ではない柱を合法手として列挙する関数
- `GameState::play`: 柱にコマを落として、次状態と最後に置かれた位置を返す関数

次に進むなら、勝敗判定で調べる「方向」を定義します。最後に置かれた `Position` から、横・奥行き・高さ・平面斜め・空間斜めの各方向に同じ色が何個つながっているかを調べる設計に進めます。
